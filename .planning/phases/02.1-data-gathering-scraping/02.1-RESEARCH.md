## Common Pitfalls

### Pitfall 1: HMO Register Column Mismatch (CRITICAL)
**What goes wrong:** `parse-hmo.js` will throw "Cannot map required columns" errors because the actual HMO register files use different column headers than the mappings expect.
**Root cause:** The column mappings in `parse-hmo.js` were written generically, but the actual files have:
- Manchester file (`FOI7139_HMORegister8_4_22.csv`): Uses `Licence Holder Name` (matches) and `premises address` (NO MATCH — expects `Address` or `Property Address`)
- Stockport files (`FOI_101006962152...` and `Public_Register...`): Use `Name & address of licence holder` (NO MATCH) and `Address of licenced HMO` (NO MATCH), `Post code` (matches)
**How to avoid:** Update `COLUMN_MAPS` in parse-hmo.js to include the actual column headers from these files before running.
**Warning signs:** "Cannot map required columns" error on first run.

### Pitfall 2: CCOD Download Requires Account + Licence Agreement
**What goes wrong:** CCOD cannot be downloaded via simple curl/wget. The gov.uk service requires creating an account and accepting licence terms.
**Root cause:** HM Land Registry data service has a web-based licence agreement flow.
**How to avoid:** Use Firecrawl or Playwright to navigate the download flow, or have Sam download manually and provide the file path.
**Warning signs:** 403 errors or redirect loops when trying direct download.

### Pitfall 3: Apify Actor Output Schema Varies
**What goes wrong:** `process-openrent.js` and `process-rightmove.js` expect specific JSON field names, but different Apify actors use different field naming conventions.
**Root cause:** Each Apify actor developer uses their own naming scheme.
**How to avoid:** After running an Apify actor, inspect the first few results to verify field names match what the processor scripts expect. Add field name fallbacks if needed.
**Warning signs:** All records imported with empty/null fields despite data being present.

### Pitfall 4: Database May Contain Stale Test Data
**What goes wrong:** `data/sjb-leads.db` already exists and may contain test data from Phase 1/2 development.
**Root cause:** Dev testing inserted mock/test records.
**How to avoid:** Check row counts before and after each import. Consider whether to wipe and recreate the DB or just run with INSERT OR IGNORE.
**Warning signs:** Unexpected row counts, mixed test and real data in exports.

### Pitfall 5: EPC API Rate Limiting
**What goes wrong:** EPC API returns 429 if too many requests in quick succession.
**Root cause:** EPC Register has undocumented rate limits.
**How to avoid:** The script already has 1-second delays between postcodes and 30-second retry on 429. With 13 postcodes, this should be fine.
**Warning signs:** Multiple 429 retries in console output.

### Pitfall 6: Companies House Rate Limiting (600/5min)
**What goes wrong:** CH API has a documented 600 requests per 5-minute window. The script fetches officers for every company, which multiplies requests.
**Root cause:** Each company search page = 1 request, each officer lookup = 1 request.
**How to avoid:** Rate limiter is already built (550 safety threshold). For large result sets, execution may just take a while.
**Warning signs:** "Rate limit: waiting..." messages in console output.

### Pitfall 7: HMO Files Are in Phase Directory, Not data/ Directory
**What goes wrong:** `config.js` defines `HMO_DIR` as `data/hmo-registers/` but the actual files are in `.planning/phases/02.1-data-gathering-scraping/HMO registers/`.
**Root cause:** Sam placed files in the phase directory rather than the data directory.
**How to avoid:** Either copy files to `data/hmo-registers/` or pass explicit file paths to `parse-hmo.js`.

## HMO Register File Analysis

### File 1: `FOI7139_HMORegister8_4_22.csv` (Manchester, 2,228 rows)
**Headers:** `ref`, `type`, `premises address`, `ward`, `first licensed`, `granted`, `expiry`, `duration_years`, `duration_months`, `Max Permitted Occupants`, `No of Storeys Comprising the Licensed HMO`, `No of Habitable Rooms`, `EPC`, `Licence Holder Name`, `Licence Holder Address`, `Managing Agent`, `Managing Agent Address`

**Mapping needed:** `name` -> `Licence Holder Name` (works), `address` -> `premises address` (NEEDS ADDING), `postcode` -> extract from address (no standalone column)

### File 2: `FOI_101006962152_HMO_public_register.csv` (Stockport, 188 rows)
**Headers:** `Address of licenced HMO`, `Post code`, `Licence Ref.`, `Short description of licensed HMO`, `Commencement date of licence & duration of licence`, plus several tribunal columns, `Name & address of licence holder`, `Name & address of person managing HMO`, `Case ID`

**Mapping needed:** `name` -> `Name & address of licence holder` (NEEDS ADDING + address stripping), `address` -> `Address of licenced HMO` (NEEDS ADDING), `postcode` -> `Post code` (works), `licence` -> `Licence Ref.` (NEEDS ADDING)

### File 3: `Public_Register_of_Mandatory_HMO_Licences_doc.csv` (Stockport older, 44 rows)
**Same headers as File 2** — same mapping applies.

### Required Column Mapping Updates for parse-hmo.js
```javascript
// Add these to COLUMN_MAPS:
manchester: {
  name: ['Licence Holder Name', 'Licence Holder', 'Landlord Name', 'Applicant Name'],
  address: ['premises address', 'Address', 'Property Address', 'HMO Address'],
  postcode: ['Postcode', 'Post Code', 'Property Postcode'],  // Manchester file has no postcode column — extract from address
  licence: ['Licence Number', 'HMO Licence', 'Licence Ref'],
},
stockport: {
  name: ['Name & address of licence holder', 'Licence Holder', 'Landlord Name', 'Licence Holder Name'],
  address: ['Address of licenced HMO', 'Address', 'Property Address', 'Address of HMO'],
  postcode: ['Post code', 'Postcode', 'Post Code', 'Property Postcode'],
  licence: ['Licence Ref.', 'Licence Number', 'HMO Licence', 'Licence No', 'License Number'],
},
```

**Special handling needed for Stockport:** The `Name & address of licence holder` column contains both name AND address combined (e.g., "Mr Simon Farrell 22a Crossefield Road, Cheadle, SK8 5PE"). Need to split name from address.

**Special handling needed for Manchester:** No standalone postcode column. Must extract postcode from the `premises address` field using regex.

## CCOD Download Strategy

The CCOD dataset is available at `https://use-land-property-data.service.gov.uk/datasets/ccod`. The download process requires:

1. Creating a free account on the Land Registry data service
2. Accepting the licence terms
3. Downloading the CSV file (~500MB)

**Options for automated download:**
- **Option A (Recommended):** Ask Sam to download manually and place at `data/CCOD_FULL.csv`
- **Option B:** Use Firecrawl to navigate the download flow
- **Option C:** Use Playwright to automate the browser-based download

Given this is a one-time download for the initial campaign, Option A is simplest and most reliable.
