## Gaps Summary

Two gaps block full goal achievement:

**Gap 1 — Empty high-priority leads CSV (blocking for one stated deliverable)**

The phase goal explicitly names "high-priority-leads" as a required output. The file exists but contains zero data rows. Root cause: EPC properties were imported without any landlord linkage. The EPC API does not return a landlord identifier, so `landlord_id` is NULL on all 9,639 EPC property records. The `build-lead-list.js` high-priority query correctly requires both a landlord name AND a poor EPC rating on an associated property — but since no EPC property is linked to any landlord, no rows satisfy this condition.

Resolution requires address-based matching to link EPC properties to landlord records from other sources (CCOD, HMO, Companies House) by matching property addresses. This was explicitly deferred to Phase 3 in the SUMMARY.md. The gap is real but the deferral was a conscious decision.

**Gap 2 — Dedup column name discrepancy (minor, non-blocking)**

Plan 04 acceptance criteria and the dedup PLAN interfaces specified `dedup_group` as the column name. The actual database schema (and `dedup.js` implementation) uses `match_group_id`. Dedup functioned correctly — 1,094 records assigned to 105 match groups. However, any script or query that references `dedup_group` (e.g. the plan's own verification command) would fail with `SqliteError: no such column`. This is documentation/naming debt, not a functional failure.
