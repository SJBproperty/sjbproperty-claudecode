<objective>
Build the EPC Register API scraper that extracts D-G rated rental properties from target postcodes and stores them in the properties table, plus a Playwright fallback scraper.

Purpose: EPC data identifies the TARGET PROPERTIES with compliance pressure (poor energy ratings). These properties are the primary hook for outreach — "your property has an EPC rating of F, new regulations mean you need to act." The EPC API does NOT return landlord names — property-to-landlord linking happens in Phase 2 via Land Registry CCOD.

Output: Working epc-api.js that populates the properties table, scrape-epc.js as fallback, integration tests.
</objective>

Properties table schema:
```sql
CREATE TABLE IF NOT EXISTS properties (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  landlord_id INTEGER REFERENCES landlords(id),
  uprn TEXT,
  address TEXT NOT NULL,
  postcode TEXT NOT NULL,
  current_energy_rating TEXT CHECK(current_energy_rating IN ('A','B','C','D','E','F','G')),
  property_type TEXT,
  tenure TEXT,
  transaction_type TEXT,
  lodgement_date TEXT,
  certificate_number TEXT,
  source TEXT NOT NULL,
  source_ref TEXT,
  scraped_at TEXT NOT NULL,
  created_at TEXT DEFAULT (datetime('now')),
  UNIQUE(uprn),
  UNIQUE(certificate_number)
);
```

    Create `scripts/epc-api.js`:

    1. Require: `dotenv` (config at top), `https` (Node built-in), `./lib/db`, `./lib/config`
    2. Auth: `const AUTH = Buffer.from(process.env.EPC_EMAIL + ':' + process.env.EPC_API_KEY).toString('base64')`
    3. `searchEPC(postcode, searchAfter)` function:
       - Build URL: `${EPC_API_BASE}?postcode=${postcode}&energy-band=d,e,f,g&size=5000`
       - If searchAfter provided, add `&search-after=${searchAfter}`
       - HTTPS GET with headers: `Accept: application/json`, `Authorization: Basic ${AUTH}`
       - Return `{ rows: parsed.rows, nextPage: res.headers['x-next-search-after'] || null }`
       - Handle HTTP errors (401 = bad auth, 429 = rate limited, 500 = server error) with console.error and retry after delay
    4. `filterRental(rows)` function:
       - Filter: `row.tenure === 'Rented (private)'` — EXACT string match
       - Do NOT filter on transaction-type alone (see Pitfall 2 in RESEARCH.md)
    5. `deduplicateByUPRN(rows)` function:
       - Group by UPRN (or by address+postcode if UPRN missing)
       - Keep only the row with the latest `lodgement-date`
    6. `insertProperties(rows)` function:
       - Use db.prepare with INSERT OR IGNORE
       - Map API fields to DB columns: `lmk-key` -> certificate_number, `current-energy-rating` -> current_energy_rating, `property-type` -> property_type, `lodgement-date` -> lodgement_date
       - Set source='epc_api', scraped_at=new Date().toISOString()
       - Wrap in db.transaction for bulk insert
    7. `scrapeAll()` main function:
       - Accept optional postcode list from CLI args: `node scripts/epc-api.js SK1 SK2` (defaults to all POSTCODES from config)
       - For each postcode: call searchEPC with cursor pagination (loop while nextPage is not null)
       - Filter, deduplicate, insert
       - Print summary: `${postcode}: ${count} rental properties (D-G rated) inserted`
       - Add 1-second delay between postcodes (rate limiting courtesy)
    8. Run scrapeAll() when script is executed directly

    - Verify field mapping from API response format (hyphenated keys) to DB format (underscored columns)

    CRITICAL: The EPC API returns hyphenated field names (e.g., `current-energy-rating`, `property-type`, `lodgement-date`, `lmk-key`). The database uses underscored column names. The mapping MUST be explicit in the code.

    CRITICAL: Use `search-after` cursor pagination, NOT `from` offset pagination (deprecated, capped at 10,000). Read `x-next-search-after` response header (lowercase — Node.js normalises headers to lowercase).

    CRITICAL: Auth is Basic Auth with Base64 of `email:apikey`, not just the API key.
