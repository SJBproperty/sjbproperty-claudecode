<decisions>
## Implementation Decisions

### Project structure
- Flat layout: `scripts/` for all JS files, `data/` for database and CSV exports
- `data/` directory gitignored — database and exports stay local only (PII protection, public repo)
- Scripts are git-tracked, data is regenerated by running them
- Plain JavaScript (Node.js) — no TypeScript, no build step
- `.env` file in project root for API keys, gitignored

### Database design
- Separate normalised tables: `landlords` table and `properties` table linked by `landlord_id`
- One landlord record, many properties — clean foundation for Phase 2 deduplication
- Per-record source provenance — each record stores which source it came from and when scraped
- Phase 1 schema only — create `landlords` and `properties` tables now; `leads`, `scores`, `contacts` tables added in later phases
- SQLite database stored at `data/sjb-leads.db`

### EPC data scope
- Core fields only (~10 fields): address, postcode, UPRN, current energy rating (A-G), tenure type, landlord name, lodgement date, certificate number, property type, transaction type
- Latest certificate only per property (by lodgement date) — no historical tracking
- Strict rental filter: tenure = 'rental' only — no unknown/blank tenure included
- EPC ratings D-G captured (compliance pressure signal)

### API key management
- `.env` file with `dotenv` package — standard approach
- Both API keys (EPC Register, Companies House) already registered and ready
- No registration steps needed in the plan

### Claude's Discretion
- Exact SQLite column types and constraints
- Error handling patterns in scripts (retry logic, rate limiting)
- Script CLI interface (arguments vs config file for postcode lists)
- MCP server configuration details
- Playwright fallback scraper implementation specifics

</decisions>
