# Phase 1: Data Foundation - Research

**Researched:** 2026-03-28
**Domain:** Government API integration (EPC Register, Companies House), SQLite database, MCP tooling
**Confidence:** HIGH

## Summary

Phase 1 builds the data infrastructure for the landlord lead generation pipeline: a SQLite database, scripts to pull data from the EPC Register API and Companies House API, and MCP server configuration for Claude Code orchestration. The tech stack is straightforward -- plain Node.js scripts with `better-sqlite3` for the database and `dotenv` for credentials, all running on Node v24.

**Critical finding: The EPC Register API does NOT return landlord names.** The ROADMAP and success criteria reference "landlord names and addresses" from the EPC API, but the domestic certificates endpoint only returns property data (address, postcode, UPRN, energy rating, tenure, property type, etc.). Landlord/owner identity must come from other sources (Land Registry CCOD in Phase 2, or Companies House cross-referencing). The EPC data's value is in identifying D-G rated rental properties in target postcodes -- the "who owns it" question is answered by Companies House and Land Registry data, not EPC data.

**Primary recommendation:** Build EPC scraper to identify target properties (D-G rated, rental tenure, target postcodes), Companies House scraper to identify property SPV landlords by SIC code, then link them via address/postcode matching where possible. Accept that full landlord identification is a Phase 2 concern when Land Registry CCOD data arrives.

## Architecture Patterns

### Recommended Project Structure
```
scripts/
  epc-api.js           # EPC Register API scraper
  scrape-epc.js        # Playwright fallback EPC scraper
  companies-house.js   # Companies House API integration
  init-db.js           # Database schema creation
  lib/
    db.js              # Shared database connection helper
    config.js          # Shared config (postcodes, API URLs)
data/                  # gitignored
  sjb-leads.db         # SQLite database
.env                   # gitignored - API keys
```

### Pattern 1: Shared Database Connection
**What:** Single module that opens the database, enables WAL mode, and exports the connection
**When to use:** Every script that touches the database imports this
**Example:**
```javascript
// scripts/lib/db.js
const Database = require('better-sqlite3');
const path = require('path');

const DB_PATH = path.join(__dirname, '..', '..', 'data', 'sjb-leads.db');
const db = new Database(DB_PATH);
db.pragma('journal_mode = WAL');
db.pragma('foreign_keys = ON');

module.exports = db;
```

### Pattern 2: Prepared Statements for Inserts
**What:** Use `db.prepare()` and `db.transaction()` for bulk inserts
**When to use:** When inserting scraped data (hundreds/thousands of rows)
**Example:**
```javascript
const insertProperty = db.prepare(`
  INSERT OR IGNORE INTO properties (uprn, address, postcode, current_energy_rating, tenure, lodgement_date, certificate_number, property_type, transaction_type, source, scraped_at)
  VALUES (@uprn, @address, @postcode, @current_energy_rating, @tenure, @lodgement_date, @certificate_number, @property_type, @transaction_type, @source, @scraped_at)
`);

const insertMany = db.transaction((rows) => {
  for (const row of rows) insertProperty.run(row);
});

insertMany(scrapedProperties);
```

### Pattern 3: Postcode Iteration with Rate Limiting
**What:** Loop through target postcodes with delay between API calls
**When to use:** Both EPC and Companies House scrapers
**Example:**
```javascript
const POSTCODES = ['SK1', 'SK2', 'SK3', 'SK4', 'SK5', 'SK6', 'SK7', 'SK8', 'M14', 'M19', 'M20', 'M21', 'M22'];

async function scrapeAll(postcodes) {
  for (const pc of postcodes) {
    console.log(`Scraping ${pc}...`);
    await scrapePostcode(pc);
    await sleep(1000); // Respect rate limits
  }
}
```

### Anti-Patterns to Avoid
- **Opening a new database connection per query:** Open once in `lib/db.js`, import everywhere
- **Async SQLite calls:** `better-sqlite3` is synchronous by design -- do not wrap in async/Promise
- **Storing API keys in scripts:** Always use `.env` via `dotenv`
- **Skipping WAL mode:** Without WAL, concurrent reads during writes will fail

## Common Pitfalls

### Pitfall 1: EPC API Does NOT Return Landlord Names
**What goes wrong:** Assuming the EPC API returns owner/landlord information. It does not.
**Why it happens:** The ROADMAP success criteria mentions "landlord names" from EPC data, but the API only returns property-level data (address, UPRN, energy rating, tenure, etc.).
**How to avoid:** Use EPC data to identify TARGET PROPERTIES (D-G rated, rental tenure). Landlord identity comes from Companies House (for Ltd/LLP owners) and Land Registry CCOD (Phase 2). The `landlords` table will be populated primarily from Companies House in Phase 1.
**Warning signs:** Empty landlord_name fields after EPC scraping.

### Pitfall 2: EPC Tenure vs Transaction Type Confusion
**What goes wrong:** Filtering on TRANSACTION_TYPE = "rental" instead of TENURE field.
**Why it happens:** Both fields relate to rental status but mean different things.
**How to avoid:** Use BOTH filters together:
- `TENURE` = "Rented (private)" -- the property IS a private rental (note: values are "Owner-occupied", "Rented (social)", "Rented (private)")
- `TRANSACTION_TYPE` = "rental" -- the EPC was lodged because of a rental transaction
- The CONTEXT.md says "tenure = 'rental' only" but the actual API field value is "Rented (private)" for private rentals
**Warning signs:** Getting owner-occupied properties in results, or getting social housing.

### Pitfall 3: EPC API Pagination Limits
**What goes wrong:** Missing results because of the 10,000 offset limit.
**Why it happens:** The `from` parameter (offset pagination) is deprecated and capped at 10,000 results.
**How to avoid:** Use cursor-based pagination via `search-after` parameter. Read the `X-Next-Search-After` response header for the next page token. Set `size=5000` for maximum page size.
**Warning signs:** Exactly 10,000 results returned for a postcode area.

### Pitfall 4: EPC API Authentication Format
**What goes wrong:** 401 errors from the EPC API.
**Why it happens:** The API uses HTTP Basic Auth where the token is Base64-encoded `email:api-key` (not just the API key).
**How to avoid:** Construct auth header as: `Authorization: Basic ${Buffer.from(email + ':' + apiKey).toString('base64')}`
**Warning signs:** 401 Unauthorized responses.

### Pitfall 5: Companies House Rate Limit (600 req / 5 min)
**What goes wrong:** Getting 429 Too Many Requests responses.
**Why it happens:** Batch querying hundreds of companies without throttling.
**How to avoid:** Implement a simple rate limiter: track request count, pause when approaching 600. With 13 postcode areas and potentially hundreds of companies per area, this is a real concern.
**Warning signs:** 429 responses, incomplete data.

### Pitfall 6: SK Postcode Granularity
**What goes wrong:** Searching for "SK" returns all of Stockport including areas outside target geography.
**Why it happens:** SK covers SK1 through SK16+ -- some areas (e.g., SK9 Wilmslow, SK10 Macclesfield, SK11 Congleton) are outside the target area of Greater Manchester/Stockport.
**How to avoid:** Use specific SK sub-postcodes: SK1-SK8 for Stockport borough core areas. Consult Sam on exact SK postcode boundaries desired. The CONTEXT says "SK" broadly but the PROJECT.md lists specific areas (Stockport, Cheadle, Marple, Bramhall, Hazel Grove, Reddish, Heaton Moor).
**Warning signs:** Properties appearing in Cheshire East or Derbyshire.

## Code Examples

### EPC API Request
```javascript
// Source: https://epc.opendatacommunities.org/docs/api/domestic
require('dotenv').config();
const https = require('https');

const EPC_EMAIL = process.env.EPC_EMAIL;
const EPC_API_KEY = process.env.EPC_API_KEY;
const AUTH = Buffer.from(`${EPC_EMAIL}:${EPC_API_KEY}`).toString('base64');

async function searchEPC(postcode, searchAfter = null) {
  const params = new URLSearchParams({
    postcode: postcode,
    'energy-band': 'd,e,f,g',
    size: '5000',
  });
  if (searchAfter) params.set('search-after', searchAfter);

  const url = `https://epc.opendatacommunities.org/api/v1/domestic/search?${params}`;

  return new Promise((resolve, reject) => {
    const req = https.get(url, {
      headers: {
        'Accept': 'application/json',
        'Authorization': `Basic ${AUTH}`,
      }
    }, (res) => {
      let data = '';
      res.on('data', chunk => data += chunk);
      res.on('end', () => {
        const nextPage = res.headers['x-next-search-after'] || null;
        resolve({ rows: JSON.parse(data).rows, nextPage });
      });
    });
    req.on('error', reject);
  });
}
```

### Companies House Advanced Search by SIC Code
```javascript
// Source: https://developer-specs.company-information.service.gov.uk/
require('dotenv').config();
const https = require('https');

const CH_API_KEY = process.env.COMPANIES_HOUSE_API_KEY;
const AUTH = Buffer.from(`${CH_API_KEY}:`).toString('base64');

// Property SIC codes:
// 68100 - Buying and selling of own real estate
// 68209 - Letting and operating of own or leased real estate (most common for BTL SPVs)
// 68320 - Management of real estate on a fee or contract basis
const PROPERTY_SIC_CODES = '68100,68209,68320';

async function searchPropertyCompanies(location, startIndex = 0) {
  const params = new URLSearchParams({
    sic_codes: PROPERTY_SIC_CODES,
    location: location,
    company_status: 'active',
    size: '100',
    start_index: startIndex.toString(),
  });

  const url = `https://api.company-information.service.gov.uk/advanced-search/companies?${params}`;

  return new Promise((resolve, reject) => {
    const req = https.get(url, {
      headers: {
        'Authorization': `Basic ${AUTH}`,
      }
    }, (res) => {
      let data = '';
      res.on('data', chunk => data += chunk);
      res.on('end', () => resolve(JSON.parse(data)));
    });
    req.on('error', reject);
  });
}
```

### Database Schema (Recommended)
```sql
-- scripts/init-db.js schema
CREATE TABLE IF NOT EXISTS landlords (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  entity_type TEXT CHECK(entity_type IN ('ltd', 'llp', 'individual', 'unknown')) DEFAULT 'unknown',
  company_number TEXT,              -- Companies House number (NULL for individuals)
  registered_address TEXT,
  sic_codes TEXT,                   -- Comma-separated SIC codes
  source TEXT NOT NULL,             -- 'companies_house', 'epc', 'land_registry', etc.
  source_ref TEXT,                  -- Source-specific ID (company number, etc.)
  scraped_at TEXT NOT NULL,         -- ISO 8601 timestamp
  created_at TEXT DEFAULT (datetime('now')),
  UNIQUE(company_number)            -- Prevent duplicate companies
);

CREATE TABLE IF NOT EXISTS properties (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  landlord_id INTEGER REFERENCES landlords(id),
  uprn TEXT,
  address TEXT NOT NULL,
  postcode TEXT NOT NULL,
  current_energy_rating TEXT CHECK(current_energy_rating IN ('A','B','C','D','E','F','G')),
  property_type TEXT,               -- house, flat, bungalow, maisonette, park home
  tenure TEXT,                      -- 'Rented (private)', 'Rented (social)', 'Owner-occupied'
  transaction_type TEXT,            -- 'rental', 'marketed sale', etc.
  lodgement_date TEXT,              -- EPC lodgement date
  certificate_number TEXT,          -- EPC certificate reference (LMK_KEY)
  source TEXT NOT NULL,             -- 'epc_api', 'epc_scraper', etc.
  source_ref TEXT,                  -- Source-specific ID
  scraped_at TEXT NOT NULL,         -- ISO 8601 timestamp
  created_at TEXT DEFAULT (datetime('now')),
  UNIQUE(uprn),                     -- One record per property (latest EPC)
  UNIQUE(certificate_number)
);

CREATE INDEX idx_properties_postcode ON properties(postcode);
CREATE INDEX idx_properties_rating ON properties(current_energy_rating);
CREATE INDEX idx_properties_landlord ON properties(landlord_id);
CREATE INDEX idx_landlords_entity_type ON landlords(entity_type);
CREATE INDEX idx_landlords_company_number ON landlords(company_number);
```

## Open Questions

1. **EPC API does not return landlord names -- how to link properties to landlords in Phase 1?**
   - What we know: EPC returns property data only. Companies House returns company data only. No direct link exists between the two in Phase 1.
   - What's unclear: Whether address matching between CH registered addresses and EPC property addresses is reliable enough for Phase 1.
   - Recommendation: In Phase 1, populate `properties` table from EPC (no landlord_id yet) and `landlords` table from Companies House (no property link yet). Accept that linking them is a Phase 2 concern when Land Registry CCOD provides the ownership bridge. Properties will have `landlord_id = NULL` until then.

2. **SK postcode scope -- which SK sub-postcodes to include?**
   - What we know: SK covers Stockport (SK1-SK5), Cheadle/Bramhall (SK3, SK7, SK8), Marple (SK6), plus areas outside target (SK9-SK16 in Cheshire/Derbyshire).
   - What's unclear: Exact SK sub-postcode boundaries Sam wants included.
   - Recommendation: Default to SK1-SK8 (covers all named areas in PROJECT.md). Make postcode list configurable so it can be adjusted.

3. **Companies House `location` filter -- how precise is geographic filtering?**
   - What we know: The advanced search has a `location` parameter but the docs do not specify whether it filters by registered address postcode, trading address, or both.
   - What's unclear: Whether searching "Stockport" returns companies registered there vs companies with Stockport in their name.
   - Recommendation: Test the `location` parameter empirically. Fallback: search by SIC codes nationally, then filter results by registered address postcode in post-processing.
